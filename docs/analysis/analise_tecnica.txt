# Arquitetura Técnica — App Flutter para Relatórios e Dashboards

## 1. Visão geral da solução

O projeto será um aplicativo mobile em **Flutter/Dart** para **iOS e Android**, focado em:

- dashboards gerenciais
- relatórios dinâmicos
- autenticação de usuários
- controle de permissões
- segmentação por loja
- suporte futuro a atualização em tempo real
- performance adequada para consulta frequente de dados

## Contexto operacional

A solução atende uma rede com aproximadamente **60 lojas**, onde:

- os dados já estão centralizados em um servidor central
- cada loja possui um identificador próprio
- existem usuários com diferentes níveis de acesso
- um gerente pode responder por uma ou mais lojas
- existe necessidade de visão individual e consolidada

---

## 2. Objetivos arquiteturais

A arquitetura deve atender aos seguintes objetivos:

- manter o app escalável
- permitir evolução sem grande retrabalho
- separar claramente regras de negócio, UI e integração
- permitir cache local controlado
- facilitar testes
- suportar crescimento para dashboards mais dinâmicos
- manter segurança de acesso no backend

---

## 3. Diretriz arquitetural principal

A recomendação é usar uma arquitetura baseada em:

- **Clean Architecture simplificada**
- organização **por feature**
- separação em camadas:
  - Presentation
  - Application
  - Domain
  - Data

## Visão das camadas

### Presentation
Responsável por:

- telas
- widgets
- navegação
- gerenciamento de estado visual
- interação do usuário

### Application
Responsável por:

- casos de uso
- orquestração das regras
- coordenação entre UI e domínio
- tratamento de fluxo da tela

### Domain
Responsável por:

- entidades
- contratos de repositórios
- regras de negócio centrais
- objetos de valor

### Data
Responsável por:

- consumo da API REST
- comunicação via socket
- persistência local
- cache
- DTOs
- mapeamento entre JSON e entidades

## 7. Modelo de autorização

## Regra principal

A autorização real deve ocorrer no backend.

O app pode esconder:

- telas
- botões
- filtros
- relatórios
- dashboards

Mas o backend deve validar sempre:

- usuário autenticado
- recurso solicitado
- lojas permitidas
- escopo do perfil
- filtros permitidos

## Modelo sugerido

A autorização deve considerar:

- usuário
- perfil
- lista de lojas permitidas
- dashboards permitidos
- relatórios permitidos
- ações permitidas
- filtros permitidos

## Exemplo

Um gerente individual pode possuir:

- perfil: gerente_loja
- lojas permitidas: 5, 8 e 14
- dashboards: vendas, metas, margem
- relatórios: vendas por vendedor, fechamento diário

## 8. Estratégia de dados

## 8.1 REST como principal
REST deve ser a base para:

- autenticação
- bootstrap da sessão
- leitura de dashboards
- leitura de relatórios
- filtros
- paginação
- exportação
- preferências

## 8.2 Socket como complementar
Socket deve ser introduzido para:

- evento de dado atualizado
- aviso de dashboard atualizado
- KPI alterado
- invalidação de cache
- alerta em tempo real

## 8.3 Estratégia recomendada
No MVP:

- usar REST como padrão
- socket opcional
- sem depender de socket para funcionamento principal

---

## 9. Estratégia de cache

## Objetivo
Melhorar performance e experiência de uso sem elevar demais a complexidade.

## O que cachear no MVP
- token de sessão
- loja ativa
- preferências
- filtros recentes
- lista de dashboards
- metadados de relatórios
- últimos resultados resumidos de KPIs

## O que não fazer no MVP
- sincronização offline completa
- edição offline
- reconciliação avançada
- atualização local complexa por merge

## Regras sugeridas
- cache por chave lógica
- TTL por recurso
- invalidação por evento
- refresh manual disponível

---

## 10. Modelo de dashboards dinâmicos

A recomendação é que o backend consiga devolver metadados suficientes para o app montar dashboards sem precisar de atualização para cada ajuste simples.

## Estrutura de um dashboard
Um dashboard pode conter:

- id
- nome
- descrição
- escopo permitido
- filtros disponíveis
- widgets

## Estrutura de um widget
Um widget pode conter:

- id
- tipo
- título
- subtítulo
- fonte de dados
- endpoint lógico
- parâmetros aceitos
- posição
- tamanho
- formato
- regras de exibição

## Tipos de widget
- KPI card
- gráfico de linha
- gráfico de barra
- gráfico de pizza
- tabela
- ranking
- comparativo
- indicador percentual
